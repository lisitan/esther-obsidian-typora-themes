# Esther Inspired 回归测试规范

本文件是 Esther Inspired 的固定发布门槛。每次修改 Obsidian 或 Typora 主题后，都必须完成与改动范围相匹配的回归测试；创建 Release、更新官方投稿或触发 Obsidian 社区扫描前，必须完成全部 P0 项目。

实际视觉测试统一使用 [`examples/regression-showcase.md`](examples/regression-showcase.md)。

## 1. 测试记录

每次测试先复制下面的记录模板，并填写到 PR、Release 说明或本次修改记录中。

```text
版本/提交：
测试日期：
测试人：
改动范围：
Obsidian 版本：
Typora 版本：
操作系统：

文件与版本检查：通过 / 失败
Obsidian 浅色：通过 / 失败 / 不涉及
Obsidian 深色：通过 / 失败 / 不涉及
Typora Paper：通过 / 失败 / 不涉及
Typora Midnight：通过 / 失败 / 不涉及

发现的问题：
遗留风险：
最终结论：允许发布 / 禁止发布
```

## 2. 测试级别

### P0：每次发布必须执行

- 手动检查主题文件、版本号和安装包。
- Obsidian 浅色与深色的核心界面和 Markdown 组件。
- 阅读模式、实时预览、编辑聚焦、悬停和选中状态。
- 本次改动直接影响的所有组件。
- Typora 有任何样式或共用资源改动时，同时测试 Paper 与 Midnight。

### P1：较大改动时执行

- Canvas、关系图谱、Bases、PDF、打印和移动端窄屏。
- Style Settings 全部选项。
- 常用第三方插件界面。
- Windows 或 Linux 抽查。

### P2：官方投稿或大版本前执行

- 全平台完整截图。
- 新安装、覆盖安装和升级安装。
- GitHub Release 下载与解压。
- README、版本号、截图和投稿页面的一致性。

## 3. 手动文件与安装检查

不运行自动测试命令，按下面清单逐项人工确认：

- [ ] 打开 `theme.css`，确认文件内容完整，没有明显缺失的大括号或被截断的样式。
- [ ] 在 Obsidian 中重新加载主题，确认主题可以正常启用，没有整体样式失效。
- [ ] 打开 `manifest.json`，确认名称、版本号、最低 Obsidian 版本和作者信息正确。
- [ ] 对照 `theme.css` 文件头、`manifest.json`、CHANGELOG 和准备发布的版本号，确认四处一致。
- [ ] 手动解压 Typora ZIP，确认包含两套 CSS 与 `esther/` 文件夹。
- [ ] 将解压后的 Typora 文件安装到主题文件夹，确认 Paper 和 Midnight 都能出现在主题菜单中。
- [ ] 搜索本次修改内容，确认没有新增不必要的 `!important`。
- [ ] 检查本次新增的选择器，确认没有使用未经限定的 `button`、`input`、`.cm-content` 等高风险全局范围。
- [ ] 在浅色和深色模式中肉眼确认正文、次要文字、链接、按钮、提示框和通知具有清晰对比度。

## 4. Obsidian P0 测试矩阵

以下每项均在浅色和深色模式下执行。

### 4.1 工作区界面

- [ ] 左侧功能区图标正常。
- [ ] 图标悬停提示文字清晰可见。
- [ ] 顶部标签页的普通、悬停、激活和关闭状态正常。
- [ ] 文件列表的普通、悬停、激活、折叠和长文件名状态正常。
- [ ] 文件夹层级清晰，没有多余竖线、光晕或装饰。
- [ ] 左右侧栏、分隔条和滚动条正常。
- [ ] 状态栏文字与图标可读。
- [ ] 设置页面的按钮、输入框、下拉框、开关和标题正常。
- [ ] 命令面板、快速切换、菜单和搜索建议正常。
- [ ] 系统通知和插件更新进度文字可读。
- [ ] 弹窗、确认框和禁用状态可辨认。

### 4.2 Markdown 基础内容

- [ ] H1–H6 层级清晰，没有裁切或重叠。
- [ ] 正文、粗体、斜体和删除线正常。
- [ ] 荧光高亮只覆盖文字下半部分。
- [ ] 普通链接和外部链接清晰可辨。
- [ ] 有序、无序和多级嵌套列表正常。
- [ ] 连续列表项目保持紧凑，没有“一行内容、一行空白”的视觉效果。
- [ ] 实时预览中的列表缩进参考线不会被异常拉长。
- [ ] 任务列表的未完成、完成和嵌套状态正常。
- [ ] 分隔线、脚注和行内代码正常。
- [ ] 中英文、数字、URL 和长文本换行正常。

### 4.3 引用与 Callout

- [ ] 单行引用的黄色引号完整显示。
- [ ] 多行引用的引号位于正文左侧，不遮挡文字。
- [ ] 引用背景始终只有一种颜色。
- [ ] 嵌套引用没有异常缩进或溢出。
- [ ] 引用中的列表、链接和高亮正常。
- [ ] Callout 的普通、警告、成功和折叠状态正常。

### 4.4 表格

- [ ] 阅读模式表格紧凑、边框完整。
- [ ] 实时预览表格紧凑、边框完整。
- [ ] 点击单元格进入编辑后，行高不会异常撑大。
- [ ] 表头、普通行、隔行背景和悬停状态正常。
- [ ] 空单元格、长文本、英文 URL 和中英混排正常。
- [ ] 增加或删除行列后布局正常。
- [ ] 窄窗口中允许合理换行或横向滚动，不遮挡正文。

### 4.5 代码

- [ ] 行内代码颜色和背景可读。
- [ ] 阅读模式代码块的工具栏、窗口圆点和边框正常。
- [ ] 实时预览代码块与阅读模式一致。
- [ ] 编辑代码时光标清晰。
- [ ] 当前行提示可见但不过度突出。
- [ ] 长代码行可以横向滚动。
- [ ] 语言标识、复制按钮和代码高亮可读。

### 4.6 Obsidian 特有内容

- [ ] 已存在双链正常。
- [ ] 未创建双链可辨认。
- [ ] 双链别名、标题链接和块链接正常。
- [ ] 嵌入笔记和嵌入图片正常。
- [ ] 反向链接和出链的字号、行高与正文片段统一。
- [ ] 大纲、标签和笔记属性正常。
- [ ] 搜索结果与命中高亮正常。

### 4.7 状态与边界

- [ ] 普通、hover、focus、active、selected、disabled 状态均可辨认。
- [ ] 加载、成功、警告和错误提示可读。
- [ ] 80%、100% 和 125% 缩放下没有严重错位。
- [ ] 主窗口较窄时没有组件覆盖正文。
- [ ] 大图片、宽表格和长代码不会撑破页面。

## 5. Typora P0 测试矩阵

Typora 有改动时，Paper 和 Midnight 必须分别完成以下项目。

- [ ] H1–H6、正文和基础强调正常。
- [ ] 单行、多行和嵌套引用正常。
- [ ] 表格的普通、长文本、空单元格和编辑状态正常。
- [ ] 行内代码和代码块正常。
- [ ] 代码块光标、当前行、工具栏和窗口圆点正常。
- [ ] 列表、任务列表、链接、图片和脚注正常。
- [ ] 高亮只覆盖文字下半部分。
- [ ] 侧边栏、大纲、文件列表和搜索框正常。
- [ ] 源代码模式与所见即所得模式没有明显断层。
- [ ] 打印或导出 PDF 的内容完整。

## 6. Style Settings

- [ ] 未安装 Style Settings 时主题正常。
- [ ] 默认设置与主题原始效果一致。
- [ ] 正文宽度设置生效。
- [ ] 紧凑界面设置生效。
- [ ] 文件列表按钮悬停显示设置生效。
- [ ] 开关设置后，浅色和深色模式均正常。

## 7. P1 兼容性抽查

- [ ] Canvas 节点、连接线和选中状态正常。
- [ ] 关系图谱节点和标签可辨认。
- [ ] Bases/数据库表格可读。
- [ ] PDF 和附件嵌入正常。
- [ ] 打印样式没有深色背景或裁切。
- [ ] Dataview、Tasks、Kanban 等常用插件没有被全局样式误伤。
- [ ] 720px 以下窄窗口可用。
- [ ] Windows 系统字体回退后没有严重布局变化。

## 8. 高风险选择器审查

修改以下区域时必须额外检查嵌套组件：

- `.cm-content`、`.cm-scroller`、`.cm-line`
- `button`、`input`、`textarea`、`select`
- `table`、`th`、`td`
- `a`、`span`、`svg`
- `.markdown-rendered`
- `.workspace-*`

审查问题：

- [ ] 选择器是否只作用于目标区域？
- [ ] 是否会影响表格单元格中的嵌套编辑器？
- [ ] 是否会影响第三方插件里的按钮或输入框？
- [ ] 是否同时考虑浅色和深色变量？
- [ ] 是否覆盖 hover、focus、active 和 disabled 状态？

## 9. 发布门槛

满足以下全部条件才允许发布：

- [ ] 手动文件与安装检查全部通过。
- [ ] 所有 P0 项目通过。
- [ ] 本次修改对应的真实交互已亲自操作验证。
- [ ] 没有未解决的阻断问题。
- [ ] 本机安装文件与仓库文件一致。
- [ ] CHANGELOG 已记录本次变化。
- [ ] Release 资源齐全。
- [ ] Obsidian 社区扫描或 Typora 投稿只在 Release 完成后触发。

任一 P0 项失败，测试结论必须为“禁止发布”。
